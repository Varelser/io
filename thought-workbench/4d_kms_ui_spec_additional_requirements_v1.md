# 4Dナレッジマネージメントシステム UI設計仕様書 追加要件補完
## 不足要件・整理整頓・格納運用・長期保守編 / 完全版 v1

---

## 0. 本書の位置づけ

本書は、既存の以下文書を補完する追加仕様書である。

- `4Dナレッジマネージメントシステム UI設計仕様書 / 時間軸・空間操作・無限キャンバス・多分割キャンバス / 完全版 v1`

既存UI仕様では、以下の中核UIが定義されている。

- 時間軸UI（4Dタイムライン・スクラバー）
- 空間操作UI（ドラッグ＆ドロップ、ジェスチャー意味づけ）
- 無限キャンバスUI
- 多分割キャンバスUI

本書では、そこに対して不足しやすい次の領域を補完する。

- 雑多な入力を受け止める流入設計
- 格納整理整頓のための中間層設計
- 後からの束ね直し・再配置・再意味化
- 資料・URL・添付物の実務的な管理
- 死蔵を防ぐレビュー・再浮上・保守
- 重複・命名崩壊・関係崩壊を防ぐ整合性機構
- 長期運用で破綻しないための作業状態管理

本書の主題は、新しい見た目を増やすことではない。
主題は、**自由入力を止めずに、後から整理・再解釈・再構成できる運用基盤をUIに持たせること**である。

## 1. 既存UI仕様に対する不足要件の診断

### 1-1. 不足しやすい領域

既存UI仕様は、時間・空間・多視点観測に強い。
一方で、長期運用を前提にした場合、次の不足が発生しやすい。

1. どこに入れるか分からない入力を受け止める層が薄い
2. 正式配置前の仮置き・保留・未統合管理が弱い
3. 資料・URL・添付物の処理状態管理が薄い
4. 後から見直す仕組みが不足し、死蔵が起きやすい
5. 重複・類似・統合候補の管理が不足しやすい
6. 作業状態と知識状態が混ざり、意味が崩れやすい
7. 無限キャンバスが広いだけで終わる危険がある
8. 整理を補助する保存ビューやスマートな集約が薄い
9. 長期的に名称・ID・分類が崩れる危険がある
10. 将来の変換・作品化・タスク化を待機させるキューがない

### 1-2. 補完方針

本書では、上記不足に対して次の方針を採用する。

1. 最初から正しい場所に入れさせない
2. 雑に入れても後から整理できるようにする
3. 実体を動かさず、見え方で束ねる機構を強化する
4. 保留・未統合・越境・所属不能を正式状態として扱う
5. しまうだけでなく、再浮上させる仕組みを持たせる
6. 自動化は勝手に確定せず、原則として提案型にする
7. 作業中の混乱を受け止めるUIを先に用意する

## 2. 追加すべき中核レイヤー

本書で追加すべき中核レイヤーは以下である。

1. Intake Layer（流入受け皿）
2. Organization Layer（整理整頓補助）
3. Review Layer（再浮上・再確認）
4. Conversion Layer（変換待機・変換補助）
5. Material Handling Layer（資料・URL処理管理）
6. Integrity Layer（重複・命名・整合性管理）
7. Workspace Memory Layer（作業面・領域・配置の記憶）

## 3. Intake Layer（流入受け皿）

### 3-1. 目的

入力時点で正しい分類を要求すると、入力が止まる。
したがって本システムでは、正式配置より前に受け止める流入層を用意する。

### 3-2. 三層構造

流入管理は次の三層を基本とする。

- Inbox
- Staging
- Archive

#### Inbox
未整理入力を即時格納する層。

対象:
- 思いつき
- URL
- メモ
- 仮説断片
- 画像参照
- 取り込み直後の資料
- 音声メモ変換直後の文章

特徴:
- 最低限の自動属性のみ付与
- 所属未定でよい
- 深い意味づけを要求しない
- 一覧と時系列で見返せる

#### Staging
整理途中・仮配置・仮接続を行う中間層。

対象:
- 仮分類済みだが未確定
- 統合候補
- 分割候補
- 球体所属候補
- タスク化候補
- 作品化候補

特徴:
- 手動整理の主戦場
- 比較、分類、統合、変換の前処理を行う
- 正式格納前の安全地帯とする

#### Archive
使わなくなったが残す層。

対象:
- 古いノード
- 完了済み案件
- 閉じたケース
- 参照頻度が低い資料
- 凍結版

特徴:
- 検索には残る
- 日常表示では露出を下げる
- 削除ではなく退避として扱う

### 3-3. UI要件

- 左パネルに `Inbox / Staging / Archive` の切替を持つ
- 多分割キャンバス内で、それぞれを別ペイン表示可能とする
- Inbox から Sphere や Bundle へドラッグ移動可能とする
- Staging では統合・分割・変換・比較を優先表示する
- Archive は折りたたみ可能な低露出表示とする

### 3-4. 自動属性

流入時に最低限自動付与する属性は以下。

- ID
- 作成日時
- 更新日時
- 流入元
- 初期格納層（Inbox / Staging / Archive）
- 所属未定フラグ
- 未統合フラグ

## 4. 仮置き・保留・所属不能・越境の正式化

### 4-1. 必要性

「整理できていない」は異常ではなく通常状態である。
したがって仮置きや保留を非正規扱いせず、正式なUI状態として扱う。

### 4-2. 状態種別

最低限、次を区別する。

- 仮置き
- 保留
- 所属不能
- 越境
- 未統合

### 4-3. 定義

#### 仮置き
まだ意味づけしていないが、残したいもの。

#### 保留
重要性はあるが、今は処理しないもの。

#### 所属不能
どの球体にも綺麗に入らないもの。

#### 越境
複数テーマにまたがり、単独所属では意味を損なうもの。

#### 未統合
断片として存在しているが、他ノードとの関係づけが未完了なもの。

### 4-4. UI表現

- 仮置き: 薄い強調、仮ラベル
- 保留: 時計または保留マーク
- 所属不能: 外側領域に配置可能
- 越境: 複数境界にまたがる表示
- 未統合: 連結不足警告または未接続強調

### 4-5. 必須操作

- 仮置き → 正式ノード化
- 仮置き → Bundle 追加
- 保留 → 再レビュー予約
- 所属不能 → 外側領域へ移動
- 越境 → 複数Sphere所属設定
- 未統合 → 統合候補一覧表示

## 5. Smart Folder / Saved View / Dynamic Collection

### 5-1. 定義

実体を移動せず、条件で自動的に集約表示するビューを持つ。

### 5-2. 目的

本システムは多重所属・多視点が前提であるため、物理フォルダだけでは整理しきれない。
したがって、見え方としての自動集約を強くする。

### 5-3. 代表例

- 未統合ノード
- 反対意見リンク未設定
- 確信度が低いノード
- 最近更新された思想系ノード
- 資料未処理
- URL未確認
- 今週触った球体
- 作品化候補
- タスク化候補
- 放置中の重要ノード
- 根拠不足の主張
- 同名・類似名候補

### 5-4. UI要件

- 左パネルに保存ビュー一覧を持つ
- 任意の検索条件を保存ビュー化できる
- 各保存ビューに件数表示を持つ
- 保存ビューを多分割ペイン内へ開ける
- タイムライン位置に応じて件数が変化してよい

### 5-5. 初期搭載すべき保存ビュー

- 未整理Inbox
- 未統合ノード
- 要レビュー
- 最近更新
- 低確信度
- 反対意見未設定
- 資料未読
- URL要検証
- 保留中
- 越境ノード

## 6. Bundle / Dossier / Case 単位

### 6-1. 目的

Node や Sphere は知識単位・テーマ単位として強いが、横断的な作業束が必要である。
そのため、Bundle 系オブジェクトを追加する。

### 6-2. 名称候補

用途に応じて次の名称を用いてよい。

- Bundle
- Dossier
- Case
- Project Pack
- Workset

内部実装上は同系オブジェクトとして扱ってよい。

### 6-3. 役割

Bundle は以下を束ねる横断単位である。

- Node
- Sphere
- Material
- URL
- Task
- JournalDay / JournalEntry
- Snapshot

### 6-4. 想定用途

- 1人のケース管理
- 1研究テーマ
- 1作品構想
- 1制度分析案件
- 1展示設計
- 1生活課題

### 6-5. Sphere との違い

Sphere はテーマ空間・文脈空間である。
Bundle は作業束・案件束・横断的な扱い単位である。

### 6-6. UI要件

- 左パネルで Bundle 一覧を持つ
- 無限キャンバス上に Bundle 領域を仮想表示できる
- Bundle をフィルタ単位・比較単位として使える
- タイムラインで Bundle の変遷を再生できる

## 7. Review / Resurfacing Layer

### 7-1. 必要性

整理整頓は格納だけでは不十分であり、再浮上機構がないと死蔵する。

### 7-2. レビュー対象

- 長期未参照ノード
- 保留ノード
- 低確信度ノード
- 高確信度だが根拠薄いノード
- 反対意見未設定ノード
- 長期未更新Sphere
- 取り込んだまま未処理の資料
- URLだけ保存して未読のもの

### 7-3. UI要件

- 右パネルに `レビュー候補` セクションを持つ
- 左パネルに `再浮上` 保存ビューを持つ
- 期限順・放置日数順・重要度順で並べ替え可能とする
- 多分割の小ペインでも確認できる簡易一覧を持つ

### 7-4. 再浮上トリガー

- 参照なし日数
- 更新なし日数
- 保留状態継続
- 関係不足
- 資料処理停滞
- タスク未進行
- 再検証日到来

### 7-5. 初期アクション

- 今見る
- 1週間後に再表示
- 保留延長
- Bundle に送る
- Archive に退避
- 統合候補へ送る

## 8. Conversion Queue（変換待機レイヤー）

### 8-1. 目的

本システムでは多くの対象が相互変換される。
ただし、その場で確定変換を要求すると流れが止まるため、変換待機キューを持つ。

### 8-2. 想定変換

- Journal → Node
- Node → Task
- Node → Hypothesis
- Node → Definition
- Node → Work Idea
- URL → Material
- Material → Note Extract
- 複数Node → 統合Node
- Node → 分割Node

### 8-3. UI要件

- 右パネルに `変換候補` セクションを持つ
- コンテキストメニューから `変換キューへ追加` が可能
- 変換キューは一覧ビューで確認可能
- 一括処理前にレビューを挟む

### 8-4. 状態

- 未処理
- レビュー待ち
- 変換可能
- 保留
- 破棄
- 実行済み

### 8-5. 設計原則

- 自動変換より提案変換を優先する
- ロスがある変換は警告する
- 変換前後のリンク維持方針を明示する

## 9. Material / URL Handling Layer

### 9-1. 必要性

資料とURLは増えやすく、しかも処理進捗が見えないと急速に詰まる。
したがってノードとは別に、資料・URL処理状態を持つ必要がある。

### 9-2. 資料の処理状態

最低限、以下を区別する。

- 未読
- 読み途中
- 読了
- 抜粋済み
- 要約済み
- ノード化済み
- 出典確定
- 作品参照済み
- 要再確認
- 凍結

### 9-3. URLの処理状態

最低限、以下を区別する。

- 後で読む
- 要検証
- 出典候補
- 一時リンク
- 作品参考
- 失効確認待ち
- 保存済み
- ノード化済み

### 9-4. UI要件

- URL専用の軽量収集箱を持つ
- 資料一覧で処理状態バッジを表示する
- URL一覧で失効・重複・未読を区別する
- Material と Node の相互参照を右パネルで見られるようにする

### 9-5. 追加補助

- 同一URL重複警告
- 既存資料との類似候補
- 未処理資料の保存ビュー
- 引用元未確定資料の抽出

---

## 10. Integrity Layer（整合性・重複・命名管理）

### 10-1. 目的

自由度が高いほど、命名崩壊・重複蓄積・関係崩れが起きやすい。
したがって整合性を監視する補助層を持つ。

### 10-2. 最低限監視すべき項目

- 同名または類似名ノード
- 断片重複
- 孤立ノード
- 参照切れ資料
- 存在しないリンク先
- Sphere 所属不整合
- 反対意見リンクの循環異常
- URL重複
- Bundle 内重複

### 10-3. 命名補助

- 自動ID付与
- 重複タイトル警告
- タイトル候補提案
- 別名 / 同義語登録
- 略称管理
- 表記ゆれ候補提示

### 10-4. UI要件

- 右パネルに `整合性警告` セクションを持つ
- 一覧ビューで `要修復` フィルタを持つ
- 重複候補比較ペインを開けるようにする
- 統合候補を変換キューへ送れるようにする

### 10-5. 原則

- 自動修正ではなく提案優先
- 一括修正は必ずレビューを挟む
- 統合時は旧ID履歴を保持する

## 11. 知識状態と作業状態の分離

### 11-1. 必要性

1つのノードが持つ意味状態と、現在の作業進行状態は異なる。
これを混ぜると運用が崩れる。

### 11-2. 知識状態の例

- 仮説
- 主張
- 観察
- 定義
- 反証
- 問い
- 断片
- 証拠

### 11-3. 作業状態の例

- 未処理
- 整理中
- 要レビュー
- 保留
- 完了
- 公開済み
- 凍結

### 11-4. UI要件

- 右パネルで知識状態と作業状態を別行表示する
- 一覧ビューで両方を独立フィルタ可能とする
- バッジ色やアイコンは混同しない

## 12. 証拠・根拠レイヤーの追加

### 12-1. 目的

思想・観察・推論・資料依拠を区別できるようにし、後から精査可能にする。

### 12-2. 最低限区別する種別

- 体験ベース
- 観察ベース
- 文献ベース
- 推論ベース
- 二次引用
- 未検証

### 12-3. UI要件

- 右パネルで根拠種別を即時確認できるようにする
- 一覧で `未検証のみ` `文献ベースのみ` などの絞り込みを可能にする
- 低確信度かつ未検証のノードをレビュー候補へ送れるようにする

## 13. 名前付き領域 / キャンバス区画

### 13-1. 必要性

無限キャンバスは広いだけでは散乱しやすい。
したがって空間そのものに意味を与える領域管理を持つ。

### 13-2. 代表的な領域名

- 発想中
- 整理待ち
- 比較台
- 今日見る場所
- 長期保留
- 変換前処理
- 資料仕分け台
- ケース作業場
- 作品構想域

### 13-3. UI要件

- 領域に名前を付けられる
- 領域色または薄い背景差を持てる
- ミニマップ上でも領域名を表示できる
- 領域へ直接ジャンプできる
- 領域ごとに保存ビューやフィルタを紐づけられる

### 13-4. 原則

領域はページではなく、作業面内の意味的区画として扱う。

## 14. 一時テーブル / 仕分け机

### 14-1. 定義

正式配置前に雑に並べて比較・統合・重複確認を行う平面領域。

### 14-2. 用途

- 類似ノードを寄せる
- 分割候補を見る
- 統合候補を検討する
- 資料とノードを並べて抜粋整理する
- 反対意見候補を並べる

### 14-3. UI要件

- 一時テーブルは無限キャンバス上に作成できる
- 一時テーブル内では自動整列提案を利用できる
- 一時テーブルから Sphere / Bundle / Staging へ移動できる
- 処理後に領域ごと折りたたみ可能とする

## 15. 自動整列・自動分類は提案型にする

### 15-1. 原則

本システムは自由度の削減を避けるため、自動整列・自動分類・自動クラスタリングを原則として提案型にする。

### 15-2. 許可される補助

- 近接候補提案
- 類似ノード候補提示
- 仮クラスタ提案
- タグ候補提示
- Sphere 所属候補提示
- 重複候補提示

### 15-3. 禁止寄りの設計

- 勝手な全面再配置
- 無確認の一括統合
- 自動で正式分類へ確定
- UI上の意味を破壊する再配置

## 18. 追加ショートカットとバッチ操作

### 18-1. 不足しやすい操作

整理整頓系は一括操作が重要である。

### 18-2. 必須バッチ操作

## 21. 追加後の全体思想

本書の追加仕様を入れることで、本システムは次の方向に強化される。

- 綺麗に整理された知識だけでなく、雑多な流入も受け止められる
- 未統合や保留を正式な状態として保持できる
- 後から束ね直し・見直し・再解釈できる
- ノードだけでなく資料やURLの滞留も管理できる
- 無限キャンバスが散乱空間ではなく作業面になる
- 多分割キャンバスが単なる観測窓ではなく整理窓になる
- 長期運用での死蔵・重複・命名崩壊を抑えられる

---

## 22. 最終結論

既存の時間軸・空間操作・無限キャンバス・多分割キャンバス仕様は、4Dナレッジマネージメントシステムの中核UIとして妥当である。

ただし、それだけでは長期運用時に次の問題が起きやすい。

- 流入が止まる
- 未整理物が詰まる
- 資料やURLが死蔵する
- 整理途中のものを扱いにくい
- 重複や命名崩壊が進む
- 後から見直す機構が弱い

したがって、追加で重要なのは派手な新規ビューではなく、以下である。

- Intake Layer
- Smart Folder / Saved View
- 仮置き / 保留 / 所属不能 / 越境 の正式化
- Bundle / Dossier / Case
- Review / Resurfacing
- Conversion Queue
- Material / URL Handling
- Integrity Layer
- 名前付き領域
- Workspace Memory

要するに、本システムをさらに強くする鍵は、
**「最初から正しく入れさせること」ではなく、「雑に入っても後から意味づけ・整理・再配置・再浮上できること」**
にある。
